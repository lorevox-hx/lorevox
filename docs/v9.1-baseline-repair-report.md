# Lorevox 9.0 Baseline Repair тАФ Diagnosis & Patch Report

**Date:** 2026-04-07
**Scope:** Restore baseline 9.0 functionality broken by v9.1 hands-free additions
**Files modified:** `ui/js/focus-canvas.js`, `ui/js/app.js`

---

## Symptoms Reported

1. **Narrator selection does nothing** тАФ clicking "Open" on a narrator card produces no visible effect
2. **+New creation does nothing** тАФ clicking "+New" to start a new narrator shows no response
3. **Mic transcription dead** тАФ clicking mic button produces no speech-to-text output
4. **Typed Lori response dead** тАФ typing and pressing Enter produces no Lori response

---

## Root Causes Found

### RC-1: `window.recognition` and `window.isRecording` always undefined (CRITICAL)

**Mechanism:** `recognition` and `isRecording` are declared with `let` in `state.js` (line 189). In JavaScript, `let` at script top-level creates a variable in the global lexical environment but does NOT add it as a property of `window`. All code accessing these via `window.recognition` or `window.isRecording` gets `undefined`.

**Impact:** `focus-canvas.js` (inside an IIFE) referenced these 12 times via `window.` prefix:
- `_startListening()` checked `window.isRecording` тЖТ always undefined тЖТ `typeof === "undefined"` was true тЖТ `toggleRecording()` never called тЖТ mic never starts
- `_hookRecognition()` polled `window.recognition` for up to 3 seconds тЖТ always null тЖТ hook never installed тЖТ no transcript routing to Focus Canvas
- `_stopListening()` checked `window.isRecording` тЖТ always undefined тЖТ `stopRecording()` never called
- `_unhookRecognition()` checked `window.recognition` тЖТ noop тЖТ handlers never restored

**Why baseline was also affected:** Focus Canvas `_installHooks()` overrides `btnMic.onclick` to open Focus Canvas instead of direct mic toggle. So ALL mic usage routes through Focus Canvas, which is broken. Similarly, `chatInput` focus is intercepted to open Focus Canvas text mode тАФ this path still worked for typing but the mic path was completely dead.

### RC-2: `sendSystemPrompt()` silently drops when `_llmReady` is false

**Mechanism:** When the user clicks "Open" on a narrator card, `lv80SwitchPerson()` runs the full narrator load sequence and then calls `sendSystemPrompt()` to send the welcome-back greeting. If the model warmup poll hasn't completed yet, `sendSystemPrompt()` returns immediately with only a `console.warn`. The prompt is lost тАФ never retried.

**Impact:** If the user clicks a narrator before the warmup poll resolves (which can take 5-30 seconds), the narrator technically loads but Lori never speaks. The chat area is blank. From the user's perspective, "nothing happened."

### RC-3 (minor): `startRecording()` made async without caller awareness

**Mechanism:** v9.1 changed `startRecording()` to `async function startRecording()` to support `await _ensureSharedMicStream()`. Callers like `toggleRecording()` don't `await` it. This means `recognition.start()` (which happens after the await) executes slightly later than before.

**Impact:** Low тАФ `_hookRecognition()` polls with a 100ms interval for up to 3 seconds, so the timing is fine. But if `_ensureSharedMicStream()` rejects (mic permission denied), the error is silently swallowed as an unhandled promise rejection. This is not a new breakage but degrades error visibility.

## Patches Applied

### Patch 1: focus-canvas.js тАФ Remove `window.` prefix (12 occurrences)

All `window.recognition` тЖТ `recognition` and `window.isRecording` тЖТ `isRecording` in the IIFE. The IIFE's scope chain includes the global lexical environment, so bare `recognition` and `isRecording` correctly resolve to the `let`-declared variables from `state.js`.

Lines affected: 398, 412, 425, 427, 428, 430, 433, 441, 442, 445, 446.

### Patch 2: app.js тАФ Expose on `window` as safety net

- `_ensureRecognition()`: after `recognition = new SR()`, added `window.recognition = recognition`
- `startRecording()`: added `window.isRecording = true`
- `stopRecording()`: added `window.isRecording = false`

Belt-and-suspenders measure тАФ ensures any future code (or third-party scripts) referencing `window.recognition` / `window.isRecording` also works.

### Patch 3: app.js тАФ Queue pending system prompt during warmup

- New variable `_pendingSystemPrompt` (declared at top of readiness gate section)
- `sendSystemPrompt()`: when `_llmReady` is false, stores the instruction in `_pendingSystemPrompt` instead of dropping it
- `_onModelReady()`: after opening the narrator switcher, checks for a pending prompt and flushes it with a 600ms delay

This ensures narrator opens that happen before model warmup completes still deliver the welcome-back greeting once the model is ready.
