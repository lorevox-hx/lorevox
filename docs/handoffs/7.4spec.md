# Lorevox 7.4 — Architecture Specification

Version: 7.4-draft
Date: 2026-03-20
Status: Pre-implementation spec
Depends on: 7.3 complete (offline UI, runtime71, cognitive-auto v7.2)

---

## Purpose

This document defines the complete implementation of Lorevox 7.4: the integration of the visual affect engine into the runtime71 conversational pipeline.

In 7.3, the camera path and the conversational path are separate systems. `emotion.js` runs MediaPipe, derives an affect state, and fires a callback — but that callback goes nowhere in the conversational pipeline. `buildRuntime71()` uses a synthetic `affectState` derived from the cognitive mode lookup table, not real facial data. Lori cannot yet reason from what she sees.

7.4 closes this gap.

---

## Phasing

### 7.4A — Reliable visual input + runtime bridge

The local server and the affect bridge ship together. If camera input is unreliable on `file://`, the bridge appears to work while silently falling back to synthetic affect — a bad partial success with no visible failure signal. The current UI is already fully local at the asset level (`lori7.3.html` loads local Tailwind and local MediaPipe; `emotion.js` resolves assets from `vendor/mediapipe/face_mesh/`). The remaining reliability issue is launch environment, not CDN dependencies.

- Local loopback server as standard launch path (`lorevox-serve.py`)
- COOP/COEP headers for SharedArrayBuffer / multi-threaded WASM
- `affect-bridge.js` (new file)
- `state.session.visualSignals` (new state)
- Patch `emotion-ui.js` to feed the bridge
- `buildRuntime71()` prefers real visual signals, falls back to synthetic
- `prompt_composer.py` consumes real visual signal + affect directives
- Retire deprecated affect POST side-channel in `emotion.js`

### 7.4B — Onboarding + baseline

Lori introduces herself and learns how to be with this person. Calibration happens invisibly inside that natural opening — not as a named technical step.

- `state.session.affectBaseline` and `state.session.onboarding` (new state)
- Lori onboarding script (greeting → capabilities → camera offer → warm-up → interview)
- Separate consent paths: camera-for-pacing and profile-photo are independent toggles
- Baseline capture during warm-up conversation window
- Baseline summary stored; raw samples discarded after finalization
- Profile photo capture function — fully decoupled from affect engine

### 7.4C — Multimodal fusion

Only after 7.4A and 7.4B are stable. This is a tuning and validation phase, not a prerequisite for real affect in the pipeline.

- Fuse visual signals into `cognitive-auto.js` `inferSignals()`
- Explicit weighting policy: visual accelerates transitions, text retains veto authority
- Thresholds tuned against real user baseline data
- Validation against false positives on aging faces

## New State Schema

Add to `state.session` in `state.js`:

```js
// state.session additions for 7.4

visualSignals: {
  affectState:      null,   // string: steady|engaged|reflective|moved|distressed|overwhelmed
  confidence:       0,      // float 0–1
  gazeOnScreen:     null,   // bool|null — OPTIONAL in 7.4A; present for forward compatibility;
                            // may remain null throughout 7.4A; required for full gaze
                            // disengagement detection in 7.4C
  blendConfidence:  0,      // float 0–1 (MediaPipe landmark confidence)
  timestamp:        null,   // ms epoch — used for stale-signal check in buildRuntime71()
},

affectBaseline: {
  active:           false,  // currently collecting baseline samples
  established:      false,  // baseline is ready; directives only fire when this is true
  startedAt:        null,   // ms epoch
  samples:          [],     // raw affect events during calibration window; cleared after finalization
  summary:          null,   // { neutralAffect, sampleCount, capturedAt }
                            // NOTE: 7.4B baseline is a coarse session-normal affect summary only.
                            // It does not attempt full facial normalization. That is a 7.4C concern.
},

onboarding: {
  complete:              false,
  cameraForPacing:       false,  // user consented to pacing camera
  profilePhotoEnabled:   false,  // user consented to profile photo
  questionsAsked:        false,  // user had opportunity to ask questions
},
```

**Important**: these three objects are independent. Consent for pacing camera, consent for profile photo, and calibration state must not be conflated. A user may want a photo without affect sensing, or affect sensing without a saved photo.

## New File: `ui/js/affect-bridge.js`

### Responsibility
Translate browser emotion events (from `emotion.js` / `emotion-ui.js`) into `state.session.visualSignals`. This is the connective tissue between the camera path and the runtime path.

### Interface

```js
window.AffectBridge74 = {
  consume(event, extras = {}),       // called from emotion-ui.js on each affect event
  beginBaselineWindow(),             // start accumulating calibration samples
  finalizeBaseline(),                // compute and store baseline summary
  getVisualSignals(),                // return current state.session.visualSignals
  isBaselineEstablished(),           // bool
  reset(),                           // clear signals and baseline (session end)
};
```

### `consume()` implementation

```js
function consume(event, extras = {}) {
  const signals = {
    affectState:     event.affectState,
    confidence:      event.confidence ?? 0,
    gazeOnScreen:    extras.gazeOnScreen ?? null,
    blendConfidence: extras.blendConfidence ?? event.confidence ?? 0,
    timestamp:       Date.now(),
  };

  // Write into shared state
  if (window.state && window.state.session) {
    window.state.session.visualSignals = signals;
  }

  // Accumulate baseline samples if window is open
  if (window.state.session.affectBaseline.active) {
    window.state.session.affectBaseline.samples.push(signals);
  }
}
```

### `finalizeBaseline()` implementation

```js
function finalizeBaseline() {
  const baseline = window.state.session.affectBaseline;
  const samples = baseline.samples;

  if (samples.length < 5) {
    // Not enough data — mark as not established, do not store bad baseline
    baseline.active = false;
    return;
  }

  // Mode of affect states during calm window
  const stateCounts = {};
  samples.forEach(s => {
    stateCounts[s.affectState] = (stateCounts[s.affectState] || 0) + 1;
  });
  const neutralAffect = Object.entries(stateCounts)
    .sort((a, b) => b[1] - a[1])[0][0];

  baseline.summary = {
    neutralAffect,
    sampleCount: samples.length,
    capturedAt: Date.now(),
  };

  baseline.active = false;
  baseline.established = true;
  baseline.samples = []; // clear raw samples after summarising
}
```

### Baseline window duration
Target: 15–30 seconds of low-demand conversation. The onboarding warm-up script (section below) provides exactly this window. Do not use a timer alone — begin when camera consent is confirmed, finalize when the onboarding conversation reaches line 11 (transition to interview).

## Patch: `app.js` — `buildRuntime71()`

Current: `affect_state` is derived from the `normalizeLoriState()` synthetic lookup table.

7.4 change: prefer real visual signal when available, fall back to synthetic.

```js
// In buildRuntime71(), replace synthetic affect derivation with:

const vs = (state.session && state.session.visualSignals) || {};
const hasLiveAffect = vs.affectState && vs.timestamp
  && (Date.now() - vs.timestamp < 8000);  // stale after 8s

const affect_state = hasLiveAffect
  ? vs.affectState
  : (state.runtime.affectState || "neutral");

const affect_confidence = hasLiveAffect
  ? vs.confidence
  : (state.runtime.affectConfidence || 0);

// Add new visual_signals block to payload
const visual_signals = hasLiveAffect ? {
  affect_state:         vs.affectState,
  affect_confidence:    vs.confidence,
  gaze_on_screen:       vs.gazeOnScreen,
  baseline_established: !!(state.session.affectBaseline && state.session.affectBaseline.established),
  signal_age_ms:        Date.now() - vs.timestamp,
} : null;
```

Include `visual_signals` in the WebSocket payload alongside the existing runtime fields.

**Stale-signal behavioral invariant**: if `visualSignals.timestamp` is absent or older than 8 seconds, `hasLiveAffect` is false and `visual_signals` must be `null` in the payload. `prompt_composer.py` must treat a null `visual_signals` identically to camera-off — no affect directives, full fallback to synthetic affect. This is not an edge case; it is the expected state any time the camera is paused, the affect engine is slow, or the user steps away. The fallback must be clean and silent, not a degraded partial state.

## Authoritative Affect Path

The single authoritative path for visual affect data in 7.4 is:

```
emotion-ui.js (MediaPipe callback)
  → AffectBridge74.consume()
  → state.session.visualSignals
  → buildRuntime71() [prefers live signal; falls back to synthetic if stale or absent]
  → WebSocket payload (visual_signals block)
  → prompt_composer.py [affect directives gated on baseline_established]
```

The deprecated `postAffectEvent()` path in `emotion.js` is **non-authoritative** and must be removed or hard-disabled once the bridge is active. While both paths coexist, there are two independent representations of the same signal with no defined reconciliation logic. This is not acceptable. Retirement of the POST path is a 7.4A delivery requirement, not a cleanup task.

`state.runtime` remains the conversational source of truth. `state.session.visualSignals` is an input to `buildRuntime71()` — it informs the runtime, it does not replace it. There is one runtime authority: `setLoriState()` and `buildRuntime71()` remain the contract that all other systems read from. Visual signals have no direct write path to `state.runtime`.

## Patch: `prompt_composer.py`

### New payload field
`runtime71` dict now may include `visual_signals` (dict or null).

### New directive rules

Add after the existing cognitive mode directives:

```python
# v7.4 — visual affect directives (only when real signal present)
visual = runtime71.get("visual_signals") or {}
v_affect = visual.get("affect_state") or ""
v_gaze = visual.get("gaze_on_screen")      # True | False | None
v_baseline = visual.get("baseline_established", False)

if v_baseline and v_affect:
    if v_affect in ("distressed", "overwhelmed") and v_gaze is not False:
        directive_lines.append(
            "VISUAL: Narrator's facial expression indicates distress or overwhelm.\n"
            "Respond with warmth and reduced pressure. Do not stack questions.\n"
            "If distress is strong, offer a pause before continuing."
        )
    elif v_affect == "overwhelmed":
        directive_lines.append(
            "VISUAL: Narrator appears overwhelmed. Pause the interview progression.\n"
            "Offer a break, validate what has been shared, and do not advance the pass."
        )
    elif v_affect in ("reflective", "moved"):
        directive_lines.append(
            "VISUAL: Narrator appears reflective or emotionally engaged.\n"
            "Allow space. Do not rush. A brief silence or gentle acknowledgment is appropriate."
        )
    elif v_gaze is False:
        directive_lines.append(
            "VISUAL: Narrator's gaze has moved off-screen.\n"
            "Use a gentle re-engagement phrase if appropriate."
        )
```

**Critical constraint**: only apply these directives when `baseline_established` is True. Without a baseline, raw affect values from an aging face will produce false signals constantly (natural furrowed brow, drooping eyelid, reduced elasticity). Raw absolute scoring without a personal baseline is not safe for this population.

## Onboarding Script (7.4C)

This is the warm-up window that doubles as baseline calibration. It is a 7.4C concern but the script text and flow structure should be designed now so 7.4A/B code does not need to be restructured later to accommodate it.

### Framing principle
This is not "system calibration." It is "Lori meeting you."

Calibration happens inside the natural opening, not as a named step.

### Script

**Line 1 — Opening (no camera, no permissions yet)**
> "Hello. I'm Lori. I'm here to help you tell your story, at your pace. We can talk, type, pause, skip something, or come back later. Nothing has to be perfect."

*State*: `state.session.onboarding.complete = false`, `currentMode = "open"`
*Hook*: `startOnboarding74()` in `app.js` or `lori73-shell.js`; render into `#chatMessages`

---

**Line 2 — Interaction modes**
> "You can speak to me and I can listen and turn your words into text. I can also speak my replies aloud, and everything I say will stay visible on screen. If typing feels easier, that works too."

*State*: no change
*Hook*: none beyond transcript render

---

**Line 3 — Workspace**
> "As we go, I'll help build your profile, timeline, and a draft of your story with you. You can review and edit those at any time."

*State*: no change

---

**Line 4 — Camera offer (plain language)**
> "If you'd like, I can also use your camera in two optional ways. First, I can take a profile photo. Second, I can use a short warm-up moment to adjust to your lighting and expressions so I pace the conversation more gently."

*State*: present consent UI, do not start camera yet
*Hook*: `beginCameraConsent74()` — sits above existing `toggleEmotionAware()` flow

---

**Line 5 — Privacy**
> "The camera is optional. If you turn it on, it stays on this device. I don't save video, and I don't need the camera to continue."

---

**Line 6 — Questions before consent**
> "Before we begin, do you have any questions about how I work, or would you like me to explain anything again?"

*State*: `state.session.onboarding.questionsAsked = true`
*Hook*: allow free chat reply; do not advance pass; `currentPass` stays `"pass1"`

---

**Line 7 — Consent choices**
> "If you'd like to continue with the camera, I'll ask your permission now."

Visible UI choices (not bundled into one toggle):
- Use camera for gentle pacing
- Take a profile photo
- Continue without camera

*State on choice*:
```js
state.session.onboarding.cameraForPacing     = true|false
state.session.onboarding.profilePhotoEnabled = true|false
```
*Hook*: new onboarding consent UI feeds into existing FacialConsent gate in `emotion-ui.js`; does not replace it

---

**Camera start (system event, not spoken)**

*State*:
```js
state.session.affectBaseline.active    = true
state.session.affectBaseline.startedAt = Date.now()
```
*Hook*: call existing `startEmotionEngine()`; `AffectBridge74` begins consuming events

---

**Line 8 — Warm-up start**
> "Thank you. Let's take a short moment to get comfortable. You don't need to do anything special — just look toward the screen naturally if that feels comfortable."

*State*: baseline sampling active
*Hook*: `AffectBridge74.beginBaselineWindow()`

---

**Lines 9a–9c — Baseline conversation window**
> "Is this a good time to begin?"
> "How would you like me to address you?"
> "Would you like me to speak more slowly, or is this pace comfortable?"

*State*: baseline samples accumulating; name preference → `state.profile.basics.preferred` if supplied
*Note*: this is the ideal low-demand window; 15–30 seconds of calm interaction is sufficient

---

**Baseline established (system event)**

*State*:
```js
state.session.affectBaseline.established = true
state.session.affectBaseline.active      = false
state.session.affectBaseline.summary     = { ...computed }
```
*Hook*: `AffectBridge74.finalizeBaseline()`; from this point, visual directives in `prompt_composer.py` are eligible

---

**Line 10 — Profile photo (if enabled)**
> "When you're ready, I can take a photo for your profile. You can keep it, retake it, or skip it."

*State*: photo reference → profile photo field if accepted
*Hook*: photo capture function completely separate from affect engine

---

**Line 11 — Transition to interview**
> "Whenever you're ready, we can begin at the beginning. I'll start by helping place your story in time."

*State*: `state.session.onboarding.complete = true`; `currentPass` remains `"pass1"` until DOB/birthplace confirmed
*Hook*: call existing `ivStart()` or equivalent

## Cognitive-Auto Fusion (7.4C)

After baseline is established, `inferSignals()` in `cognitive-auto.js` should incorporate visual signals alongside text signals.

### Fusion policy

Visual signals can **accelerate** a mode transition but not **cause** one alone. Text retains veto power. A strong explicit verbal statement ("I'm fine," "let's keep going") overrides a mild visual signal. Visual signals only override text when they are strong, sustained (multiple consecutive events), and the text is ambiguous or absent.

```js
// In inferSignals(), add after existing text-based detection:

const vs = (state.session && state.session.visualSignals) || {};
const baselineOk = state.session.affectBaseline &&
                   state.session.affectBaseline.established;
const signalFresh = vs.timestamp && (Date.now() - vs.timestamp < 8000);

if (baselineOk && signalFresh) {
  if (vs.affectState === "distressed" && vs.confidence > 0.75) {
    signals.distress = true;
  }
  if (vs.affectState === "overwhelmed" && vs.confidence > 0.80) {
    signals.distress = true;
    signals.fatigue  = true;
  }
  if (["reflective", "moved"].includes(vs.affectState) && vs.gazeOnScreen === false) {
    // Drifting — gently flag confusion without escalating
    signals.confused = signals.confused || (confusionTurnCount > 1);
  }
}
```

## Local Server Bootstrap (7.4A)

### Target
Replace `file://` as the supported default with `localhost:8000`.

### Minimal launch script

`lorevox-serve.py` in project root:

```python
#!/usr/bin/env python3
"""
Lorevox local server — required for camera + SharedArrayBuffer support.
Run: python lorevox-serve.py
Then open: http://localhost:8000/ui/lori7.3.html
"""
import http.server, socketserver, os

class LorevoxHandler(http.server.SimpleHTTPRequestHandler):
    def end_headers(self):
        # Required for SharedArrayBuffer (multi-threaded WASM)
        self.send_header("Cross-Origin-Opener-Policy",   "same-origin")
        self.send_header("Cross-Origin-Embedder-Policy", "require-corp")
        http.server.SimpleHTTPRequestHandler.end_headers(self)

os.chdir(os.path.dirname(os.path.abspath(__file__)))
PORT = 8000
with socketserver.TCPServer(("", PORT), LorevoxHandler) as httpd:
    print(f"Lorevox running at http://localhost:{PORT}/ui/lori7.3.html")
    httpd.serve_forever()
```

### Affect on MediaPipe performance
With COOP/COEP headers, `SharedArrayBuffer` is available and MediaPipe can use the multi-threaded WASM execution path. Cross-origin isolation enables higher-performance inference and is expected to improve reliability and throughput on supported hardware. Without it, MediaPipe runs single-threaded and may drop below 15 FPS on older Intel hardware — insufficient for consistent affect sensing. Actual gains will vary by machine; validate with runtime logs on target devices before drawing conclusions.

### Retire the deprecated POST
Once `affect-bridge.js` is live and visual signals flow through `buildRuntime71()`, the `postAffectEvent()` call in `emotion.js` should be removed or guarded. It is a parallel path to the same data arriving twice in different forms. Its removal is part of **7.4A**, not deferred — leaving it active while the bridge is also active creates ambiguous dual authority over the same signal.

```js
// In emotion.js postAffectEvent() — replace body with:
// Deprecated in 7.4A. Affect now flows through AffectBridge74 → state.session.visualSignals
// → buildRuntime71() → prompt_composer.py.
// console.log('[LoreVoxEmotion] postAffectEvent deprecated — affect bridge active');
```

## Non-Regression Requirements

If the user declines camera access, Lorevox must behave exactly as it does in 7.3. This is a hard requirement, not a preference.

7.4 must not break:

- Camera-off / text-only behavior — synthetic affect path must remain fully functional
- Voice-only use
- pass / era / mode propagation through runtime71
- `ivStart()` and tab switching behavior
- Existing `FacialConsent` gate in `emotion-ui.js` — onboarding feeds into it, does not replace it
- Offline operation — `lorevox-serve.py` serves only local assets; no external calls
- 7.3 regression test: all 7.3 acceptance criteria must still pass under 7.4

Text-based cognitive mode decisions remain the primary authority. Visual signals inform and may accelerate transitions; they do not override clear verbal input. A user who declines the camera gets the full Lorevox experience.
