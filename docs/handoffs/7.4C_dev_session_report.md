## Bug Found and Fixed: `window.state` Guard Pattern

**Root cause**

`state` is declared as `let state = {...}` at the top level of `state.js`. Because it uses `let` (not `var`), it does not attach to `window`. Any guard written as `if (!window.state) return` silently fails — the function exits immediately even though `state` is fully defined and accessible to all scripts in the page.

**Files patched**

Three files contained this pattern. All were corrected to `typeof state === "undefined"`:

| File | Location | Old guard | New guard |
|------|----------|-----------|-----------|
| `ui/js/app.js` | `startOnboarding74()` | `if (!window.state)` | `if (typeof state === "undefined")` |
| `ui/js/app.js` | `beginCameraConsent74()` | `if (!window.state \|\| ...)` | `if (typeof state === "undefined" \|\| ...)` |
| `ui/js/app.js` | `finalizeOnboarding74()` | `if (!window.state \|\| ...)` | `if (typeof state === "undefined" \|\| ...)` |
| `ui/js/affect-bridge.js` | `ensureSession74()` | `if (!window.state)` | `if (typeof state === "undefined")` |

**Same pattern applies to** `cameraActive` and `emotionAware` — both are `let` declarations in `state.js` and `emotion-ui.js` respectively. These are shared across all non-module scripts in the same document, so `window.cameraActive` would also fail. Any future guard should use `typeof x === "undefined"` rather than `!window.x`.

## Additional Fix: Browser Cache

The `affect-bridge.js` fix did not take effect on the first reload because the browser had a cached version of the file from before the patch. A hard reload (`Ctrl+Shift+R`) was required. This is expected in local dev but worth noting: the `lorevox-serve.py` server does not set `Cache-Control: no-cache` headers on JS files. Adding that would remove this friction during future dev sessions.

## Known Issue — MediaPipe WASM Crash

**Symptom**

Every time the camera starts, the following exception fires inside the vendored MediaPipe bundle:

```
RuntimeError: abort(undefined) at SolutionWasm$loadGraph
  at face_mesh_solution_simd_wasm_bin.wasm : func7263
```

**What this means**

The camera stream itself starts successfully (`cameraActive: true`, preview shows live video). The crash happens when MediaPipe's WASM runtime tries to load the FaceMesh computation graph — immediately after the stream opens. Because of this, no `onBrowserAffectEvent` callbacks ever fire, which means:

- `state.session.visualSignals` stays null
- `state.session.affectBaseline.samples` stays at 0
- `buildRuntime71()` always returns `visual_signals: null`
- The affect path is open and waiting but receives nothing

**This is a deployment issue, not a JavaScript issue.** The 7.4C pipeline is architecturally correct. The WASM loader is failing before it processes a single frame.

**Two likely causes**

1. **Corrupted or missing `.binarypb` graph file.** MediaPipe's FaceMesh requires a binary protobuf graph file (`face_mesh.binarypb` or similar) in the vendor folder. If this file is missing, truncated, or was not copied correctly, `loadGraph` will abort.

2. **SIMD WASM not available.** The vendored file is `face_mesh_solution_simd_wasm_bin.wasm` — the SIMD-accelerated variant. If Chrome's WASM engine does not have SIMD enabled for this context, the binary will fail to initialise. A non-SIMD fallback (`face_mesh_solution_wasm_bin.wasm`) may be needed.

**Diagnostic steps**

```bash
# Check vendor folder contents and file sizes
ls -lh ui/vendor/mediapipe/face_mesh/

# Any zero-byte or unexpectedly small files are the likely culprit
# Expected: face_mesh.binarypb (several hundred KB), both .wasm files (several MB)
```

Also check `emotion.js` for the `locateFile()` patch — it routes WASM loading to the local vendor path. If the path mapping is wrong, the WASM binary loads but the graph file is not found.

## New Feature Added: Camera Preview Float

A draggable camera preview popup was added as part of this session. It is wired into `beginCameraConsent74()` and auto-appears once the camera is active.

**Files changed:**
- `ui/css/lori73.css` — preview container, title bar, close button, re-open pill styles
- `ui/js/lori73-shell.js` — `showCameraPreview()` and `_attachPreviewStream()` functions added to `window.lv74`
- `ui/js/app.js` — `beginCameraConsent74()` calls `lv74.showCameraPreview()` after baseline window opens

**Behaviour:**
- 180×135px floating video, mirrored (natural self-view)
- Default position: top-centre of viewport
- Drag from the title bar to reposition
- ✕ button hides the preview — camera continues running
- `📷 Camera` pill appears at top-centre when preview is hidden
- Click the pill to bring the preview back
- Can also be called manually: `lv74.showCameraPreview()`
