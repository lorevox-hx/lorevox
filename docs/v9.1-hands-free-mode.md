# v9.1 Hands-Free Mode

**Date:** April 2026
**Files changed:** `ui/js/app.js`, `ui/js/focus-canvas.js`, `ui/lori9.0.html`
**Status:** Patch drafted, ready for testing after integration


## What This Is

Phase 1 of a hands-free upgrade to make Lorevox usable by older adults without clicking or typing during an interview. When hands-free mode is toggled on, Lori asks a question, speaks it aloud, then automatically re-opens the microphone. The narrator speaks naturally, and after two seconds of silence the system sends the message on its own. No buttons, no "send" commands.

This is scoped to the browser-side Chrome/Web Speech API path. It does not change the server, the TTS engine, the LLM pipeline, or the WebSocket chat. Everything added is gated behind a `handsFreeMode` toggle so existing manual-mic users are unaffected.


## What Changed

### 1. Auto-resume listening after TTS (`app.js`)

The `drainTts()` function already stops the microphone before Lori speaks and clears the `isLoriSpeaking` flag in a `finally` block. v9.1 adds a 400ms delayed call to `startRecording()` inside that same `finally` block, gated on `handsFreeMode === true`.

The 400ms delay exists to let the browser's Acoustic Echo Cancellation (AEC) and automatic gain control settle after TTS playback ends. Without this buffer, the microphone can pick up residual speaker output and trigger false recognition results.

The existing STT guard contract is preserved: `recognition.onend` only auto-restarts when `isRecording === true && isLoriSpeaking === false`, so the re-armed mic stays alive as expected.

**Key code path:**

```
drainTts() finally block
  → isLoriSpeaking = false
  → ttsBusy = false
  → if handsFreeMode && !isRecording:
      setTimeout(startRecording, 400)
```


### 2. Silence-based auto-send (`app.js`)

A new silence monitor uses the Web Audio API `AnalyserNode` to compute the Root Mean Square (RMS) of the microphone input in real time via `requestAnimationFrame`. When the RMS stays below 0.01 (1% of max amplitude) for 2000ms continuously, the system auto-sends whatever text is in the chat input.

Parameters chosen for elderly speakers:

| Parameter | Value | Reason |
|---|---|---|
| `_SILENCE_THRESHOLD` | 0.01 | Filters typical domestic background noise |
| `_SILENCE_DURATION` | 2000ms | Accommodates slower speech with longer pauses |
| AnalyserNode `fftSize` | 2048 | Good frequency resolution without excessive CPU |
| `smoothingTimeConstant` | 0.3 | Responsive to speech onset/offset |

The RMS calculation:

```
RMS = sqrt( (1/n) * sum( ((sample - 128) / 128)^2 ) )
```

Where samples come from `getByteTimeDomainData()` (unsigned 8-bit, 128 = zero crossing).

The monitor only runs when `handsFreeMode` is true. It pauses during TTS playback (`isLoriSpeaking`) and stops entirely in `stopRecording()`. The silence progress (0.0 to 1.0) is exposed on `window._silenceProgress` for the Focus Canvas countdown display.

Auto-send only fires when `chatInput` has non-empty text and no send is already in flight (the monitor loop exits after calling `sendUserMessage()`, so `drainTts()` handles re-arming when Lori finishes her next response). If the input is empty when the silence timer expires, the timer resets and waits for the narrator to speak again.


### 3. Shared microphone stream (`app.js`)

Both the Web Speech API and the `AnalyserNode` need microphone access. Requesting `getUserMedia` twice can trigger a duplicate permission prompt on some browsers (tracked as a known issue in the repo).

v9.1 introduces `_ensureSharedMicStream()` which calls `getUserMedia({ audio: true })` once, builds the `AudioContext` → `MediaStreamSource` → `AnalyserNode` graph, and reuses the stream across recording sessions. The analyser is exposed as `window._micAnalyser` for focus-canvas.js to read.

`startRecording()` now calls `_ensureSharedMicStream()` before `recognition.start()`, so the mic permission is established before recognition begins.


### 4. Transcript cleanup improvements (`app.js`)

The existing `_normalisePunctuation()` regex chain was extended with conservative cleanup rules. These preserve the narrator's words (Lorevox's product principle: the person's words are primary) while fixing common speech-to-text artifacts:

- Remove space before punctuation: `"word ."` → `"word."`
- Ensure space after punctuation (except inside ellipsis or decimals)
- Collapse repeated commas/periods (but preserve `...` as ellipsis)
- Capitalize after sentence-ending punctuation (`. `, `? `, `! `)
- Capitalize standalone `"i"` → `"I"`
- Capitalize first character of the transcript

No words are changed, rewritten, or removed. This is punctuation/capitalization cleanup only.


### 5. Audio visualization in Focus Canvas (`focus-canvas.js`)

When the Focus Canvas opens in voice mode, a `<canvas>` element now shows a 7-bar audio level meter. Each bar represents a segment of the mic input buffer, with height proportional to that segment's RMS energy. Color shifts from soft blue (quiet) to bright green (speaking).

For users with `prefers-reduced-motion` enabled, the bars are replaced by a single dot that changes color with volume.

The silence countdown is shown as a progress bar below the transcript. When silence is detected in hands-free mode, it fills over 2 seconds with a label showing "Sending in 2s... 1s..." then sends. The bar resets when the narrator starts speaking again.


### 6. Hands-free toggle (`lori9.0.html`)

A small circular button next to the mic button in the footer. Uses a recycle/loop icon to suggest "continuous." Green glow when active. Calls `setHandsFreeMode(!handsFreeMode)` on click.

State is persisted to `localStorage` under the key `lorevox_handsFreeMode` and restored on page load in `lv80Init()`.

New CSS for the toggle, the audio visualization canvas, the silence progress bar, and the countdown label.


## Quick Smoke Test

1. Open `lori9.0.html` in Chrome
2. Load a narrator (e.g., Kent James Horne)
3. Click the small loop button next to the mic — it should glow green
4. Click the mic or tap the Focus Canvas to start a voice session
5. Speak a sentence and stop — after ~2 seconds of silence the message should auto-send
6. Lori responds via TTS — when she finishes speaking, the mic should auto-rearm
7. The Focus Canvas should show bouncing audio bars while you speak and a filling progress bar during silence
8. Toggle hands-free off — behavior should revert to manual mic + "send" command
9. Reload the page — the hands-free toggle should restore its last state (green if it was on)
10. While hands-free is active and no text is in the input, silence should reset the timer rather than send an empty message

If you want the minimum useful pass, run these 8: T01 (toggle persistence), T02 (manual mode unchanged), T03 (auto-send after silence), T04 (countdown reset on resumed speech), T07 (mic auto-resume after TTS), T08 (no Lori feedback loop), T13 (Focus Canvas bars + silence bar), T15 (no duplicate mic permission prompt).

## What's Deferred (Future Phases)

The following ideas were researched but intentionally left out of this phase. They are documented here for when the project is ready.


### Phase 2: Server-Side STT with Faster-Whisper

Replace the browser's Web Speech API with server-side transcription using faster-whisper (CTranslate2 reimplementation of OpenAI Whisper). The server already has an unused STT endpoint at `/api/stt/transcribe` in `server/code/api/routers/stt.py`.

**What it adds:** Better accent handling, multilingual support, and consistency across browsers. Whisper large-v3 was trained on 680,000 hours of multilingual audio and significantly outperforms Web Speech API for accented English.

**Architecture:** Browser captures raw audio via MediaRecorder, streams PCM chunks to a new WebSocket endpoint `/api/stt/stream`, faster-whisper transcribes with VAD filtering, transcript streams back. The existing silence monitor and auto-send logic from v9.1 would still apply.

**Why deferred:** Requires a new WebSocket endpoint, changes to the audio capture path, and GPU memory management for the Whisper model. The current Chrome/Web Speech API path works well enough for English-only narrators.


### Phase 3: Spanish and Italian Support

Whisper large-v3 natively supports 99 languages including Spanish and Italian. This phase enables narrators to speak in their preferred language.

**Language detection:** Whisper auto-detects from the first 30 seconds of audio. Alternatively, a language preference field in the narrator profile (people table + UI) can be passed as a hint to skip detection and reduce time-to-first-token.

**Accent comparison:**

| Language | Web Speech API | Whisper large-v3 |
|---|---|---|
| English (standard) | Good | Excellent |
| English (Southern US) | Poor-Fair | Very Good |
| English (Italian accent) | Poor | Good-Very Good |
| Spanish | Separate mode only | Native support |
| Italian | Separate mode only | Native support |
| Code-switching (Spanglish) | Fails | Reasonable |

**Punctuation map extension:** The `_normalisePunctuation()` function can be extended with language-specific spoken punctuation words:

```javascript
// Spanish: "punto" → ".", "coma" → ",", "signo de interrogacion" → "?"
// Italian: "punto" → ".", "virgola" → ",", "punto interrogativo" → "?"
```

**TTS voices:** Current Coqui VITS only has English voices. For Spanish and Italian narrators, options include Azure Cognitive Services neural voices (`es-ES-XimenaNeural`, `it-IT-ElsaNeural`) or hosting a multilingual TTS model locally. This is a separate decision from STT and involves cost vs. latency tradeoffs.

**Why deferred:** Requires Phase 2 (server-side STT) first, plus TTS voice selection, LLM prompt language handling, and narrator profile schema changes.


### Phase 4: Auto-Correction and Proper Noun Grounding

Use narrator profile data to improve transcription accuracy.

**Initial prompt seeding:** Whisper accepts an `initial_prompt` parameter that biases the model toward expected vocabulary. Construct it from the narrator's full name, family member names (from graph_persons), hometown, and key places from previous sessions. This dramatically improves proper noun recognition.

**Post-transcription correction:** After Whisper returns a transcript, run a lightweight fuzzy-match pass against a narrator-specific vocabulary list. Common substitutions like "Janice" misheard as "Janet" or "Horne" misheard as "Horn" can be caught automatically.

**Confidence display:** For new narrators, show a brief "Is this right?" confirmation before auto-sending. Once they trust the system, switch to full auto-send.

**Why deferred:** Depends on Phase 2 (Whisper provides the initial_prompt API; Web Speech API does not). The fuzzy matching also needs the relationship graph to be populated, which happens later in the interview process.


### Phase 5: Wake Word ("Hey Lori")

A local wake-word engine running in a Web Worker via WebAssembly. Options include Picovoice Porcupine (commercial, per-device licensing) or openWakeWord / Mycroft Precise (open source, slightly lower accuracy).

When the user says "Hey Lori," the Focus Canvas opens automatically and recording starts. The Lori status dot in the header would pulse yellow in standby mode to show the system is listening for the wake word.

**Why deferred:** Separate feature from hands-free STT. Adds complexity in power management (always-on mic), privacy considerations, and wake-word model training. Current flow (click mic once, then hands-free from there) is sufficient for supervised interview sessions.


### Phase 6: Intelligent Interruption ("Hush" Logic)

If the narrator starts speaking while Lori is talking, detect it via VAD after 400ms of sustained speech, immediately pause TTS playback, flush the TTS queue, and yield the turn.

**Why deferred:** Requires high-quality echo cancellation to distinguish narrator speech from speaker output. More complex than the first hands-free pass and risks false positives in noisy home environments.


## Dependencies

No new npm packages or pip packages were added in v9.1. Everything uses browser-native APIs:

- `navigator.mediaDevices.getUserMedia` (mic access)
- `AudioContext` + `AnalyserNode` (Web Audio API for RMS)
- `requestAnimationFrame` (silence monitoring and canvas drawing)
- `localStorage` (hands-free toggle persistence)
- `<canvas>` 2D context (audio visualization)
