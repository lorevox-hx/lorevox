## 4. Non-Negotiable Architecture Rule

All Lori behavior must continue to flow through the single validated path:

`state.runtime → buildRuntime71() → WebSocket/SSE → prompt_composer.py`

Do not create a second behavior pipeline.

That means:

- `state.runtime` is the only source of truth for runtime behavior
- `buildRuntime71()` is the only runtime payload builder
- `setLoriState()` is the only entry point for runtime state updates
- `prompt_composer.py` is the backend behavior gate

If the model does not receive the state, the feature does not exist.

## 12. Known Offline Risks Tonight

### Risk 1 — Tailwind CDN
If the shell still depends on:
`<script src="https://cdn.tailwindcss.com"></script>`
then a fully offline render is not guaranteed.

### Risk 2 — MediaPipe CDN
If the shell or emotion engine still loads:
- `face_mesh.js`
- `camera_utils.js`
- or `locateFile` points to jsDelivr

then camera affect mode is not offline-safe.

### Risk 3 — Camera path
Even if the rest of Lorevox works offline, enabling emotion-aware mode may fail until MediaPipe assets are vendored locally.

## 13. Minimal Hardening Needed for a True Offline Browser Test


### C. Change `emotion.js` locateFile path
It must point to local vendored assets rather than jsDelivr.

### D. Re-test with network disabled
Only then is the browser-side offline story truly complete.
